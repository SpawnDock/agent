# Agent SpawnDock

## WAL.md

- Always read `WAL.md` at the start of every session.
- Never modify `WAL.md` under any circumstances.

## Workspace Map

- This repository is a multi-project SpawnDock workspace. The root repo is coordination and documentation; the real products live under `repo/` as separate submodules/packages.
- Treat `README.md` in the root as the high-level workspace index. For implementation details, switch to the relevant subproject and read its local `README.md` and `package.json`.
- Do not assume a single app entrypoint exists at the workspace root. Most work should happen inside one specific subproject.

## Subprojects

- `repo/tma-project` is the canonical Telegram Mini App starter. It is a Next.js + TypeScript template with SpawnDock local preview support and `spawndock/*.mjs` dev scripts.
- `repo/create` is the bootstrap CLI. It clones/applies the TMA starter overlay and writes runtime files such as `spawndock.config.json`, `spawndock.dev-tunnel.json`, `opencode.json`, and project-specific manifest/env files.
- `repo/dev-tunnel` is the local preview tunnel client. It connects the local TMA dev server to the SpawnDock control plane over WebSocket.
- `repo/mcp-client` is the stdio-to-SSE MCP bridge used by AI agents. It exposes the SpawnDock knowledge server as a local MCP tool.
- `repo/api` is the backend/control-plane side. It hosts the MCP/SSE server and related backend logic used by the wider SpawnDock platform.
- `repo/cli` is an environment/automation CLI for isolated Docker-based repo work. It is adjacent tooling, not the main TMA runtime path.

## Canonical Flow

- SpawnDock development centers on `repo/tma-project` as the app template a user actually runs.
- `repo/create` prepares a project based on that template and injects SpawnDock-specific runtime configuration.
- `repo/dev-tunnel` is what makes local TMA preview reachable through SpawnDock infrastructure.
- `repo/mcp-client` is how local AI tooling talks to the SpawnDock MCP knowledge service.
- `repo/api` provides the backend services those local tools depend on.

## Working Assumptions

- When a task mentions "run the project", first determine which subproject is intended; defaulting to the workspace root is usually wrong.
- For local app startup, `repo/tma-project` is the primary candidate unless the task explicitly targets backend, tunnel, CLI, or MCP client work.
- Configuration files generated by SpawnDock bootstrap are part of the expected runtime flow, especially in `repo/tma-project`.
